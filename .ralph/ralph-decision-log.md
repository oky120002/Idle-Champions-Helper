# Ralph 决策记录

- 决策点：英雄筛选结果卡如何在不脱离现有视觉语言的前提下压缩高度并改善立绘展示
  选择方案：采用“左侧信息摘要 + 右侧头像舞台 + 顶部立绘背板”的横向卡片结构，并把属性概览改成更紧凑的两列摘要面板
  推荐理由：能在保留现有金属感与卡片层次的同时减少纵向堆叠，让名字、定位、联动队伍和属性标签更快扫读，也避免立绘被整卡背景拉伸后只剩局部

- 决策点：英雄页随机排序是否写入 URL 或筛选状态
  选择方案：沿用立绘页和宠物页的本地随机种子做法，只影响当前过滤结果与展示窗口，不写入 URL
  推荐理由：交互和已有两个随机排序入口保持一致，用户可以随时重洗当前结果，但不会污染分享链接或破坏已有筛选恢复逻辑

- 决策点：宠物图鉴页的“默认数量 + 显示全部”要对齐到什么程度
  选择方案：直接复用英雄页 / 立绘页的 50 条默认窗口模式，包含顶部状态胶囊、顶部展开按钮、底部尾部按钮，以及筛选变更时自动收起回默认窗口
  推荐理由：这样不是只补一个按钮，而是把结果窗口的节奏、文案预期和操作反馈统一下来；用户切换三个图鉴页时不会遇到不同的展开规则

- 决策点：立绘图鉴卡片标题如何同时满足“单语言显示”和“皮肤名 + 英雄本名同线展示”
  选择方案：标题改为只输出当前国际化语言；皮肤卡使用“皮肤名 · 英雄本名”的单行组合，本体卡只显示英雄名，并去掉独立的英雄名副行
  推荐理由：这样能直接消除中英双语混排带来的噪音，同时保留皮肤识别度和英雄归属信息；卡片层级更紧凑，视觉上也比双语副标题或双行堆叠更稳定

- 决策点：英雄详情页顶部可用性信息如何收口，避免商店/时间门/黑名单类字段继续占据阅读空间
  选择方案：只保留当前可用与下次活动这类高层级状态，过滤掉商店上架、时间门开放和黑名单类字段，不再把它们放进可见标签流
  推荐理由：顶部卷宗更像阅读入口而不是原始数据表，保留核心状态足够支撑判断，删掉冗余字段后也能减少跨区块重复和视觉噪音

- 决策点：英雄详情页所有日期字段的时区基准如何处理
  选择方案：统一按 UTC 日历日渲染，只输出年月日，不保留时分秒，也不依赖浏览器本地时区
  推荐理由：这样可以避免跨时区浏览时日期前后跳一天，保证页面展示与数据语义稳定一致

- 决策点：英雄详情页导航高亮应如何识别来源页面，而不是继续隐式绑定到英雄筛选
  选择方案：在详情入口显式写入 `activeNavigationTo`，导航高亮解析优先读取该字段，再回退到 `returnTo.pathname`
  推荐理由：把“来源高亮”和“返回目标”解耦，后续即使调整返回路径、查询参数或返回文案，也不容易把详情页高亮逻辑一起带坏

- 决策点：全局导航标题区如何在压缩高度的同时保留识别度和质感
  选择方案：将主标题与副标题压成同一行，副标题改为四段式动作短语，并用轻量金属感胶囊高亮承接
  推荐理由：这样能直接减少头部行高与阅读跳跃，同时保留站点定位信息；副标题缩成短标签后，也更适合在收起前后的节奏切换里保持稳定视觉重心

- 决策点：通用页面标题栏要如何同时压缩高度、保留数据版本信息，并处理右侧统计摘要区
  选择方案：将标题栏重构为“顶部信息行 + 标题主句”的双层结构；把公共数据版本与采集时间并入顶部信息行，桌面端把右侧统计摘要区提升到右上角，副标题整行移除
  推荐理由：顶部可扫描信息集中到一行后，标题栏高度能明显收口；右侧摘要不再压在标题下方，页面进入内容区更快，同时仍保留公共数据版本和页面统计的可见性

- 决策点：各页面左侧筛选框要如何真正统一标题模式、滚动条和纵向留白，而不是继续维护英雄页特例
  选择方案：将英雄页并入共享 `FilterSidebarLayout + FilterSidebarPanel`，三个页面的标题统一为“页面名 + 筛选”，并把间距、sticky 高度和滚动条样式继续收口到 `src/styles/shared/filters.css`
  推荐理由：只有把英雄页从旧 `champions-sidebar` 特例迁回共享组件，标题区、状态区、滚动条和 sticky 容器才能真正同源；顺带压缩共享面板的上下留白后，三个页面都会一起变得更贴边、更稳定
