# Idle Champions 项目说明

## 项目定位

- 本目录用于规划和实现《Idle Champions of the Forgotten Realms》辅助网站。
- 当前产品方向不是“再做一个大而全百科”，而是做一个更偏“个人成长导向的阵型决策台”的工具站。
- 第一阶段优先解决这些问题：资料查询、限制筛选、阵型编辑、候选英雄校验、方案保存。
- 不要一开始就做“全自动最优阵容求解器”或完整模拟器；先把可解释、可维护的决策闭环做出来。

## 当前仓库状态

- 目前本目录以调研文档为主，尚未落地前端工程、数据脚本或部署配置。
- 在没有用户明确要求前，不要擅自初始化大体量工程、数据库或云服务。
- 如果需要开始编码，先基于现有文档确认技术路线，再补最小可运行骨架。

## 关键文档索引

- `docs/idle-champions-research-roadmap.md`：项目价值判断、竞品拆解、MVP 边界、分阶段路线、核心数据模型。
- `docs/static-data-storage-research.md`：公共数据推荐用版本化 JSON 文件；个人数据推荐本地优先存到 IndexedDB。
- `docs/static-hosting-research.md`：静态部署以 GitHub Pages + GitHub Actions 为主，重点关注 SPA 路由和静态资源路径。
- `docs/china-static-hosting-research.md`：国内可访问性的备选方案；默认仍优先 GitHub Pages，必要时再考虑 Cloudflare Pages 或国内托管。

## 实施原则

- 先数据模型与规则表达，再页面和视觉层。
- 公共游戏数据优先走静态文件方案，避免在需求未稳定前引入服务端数据库。
- 个人数据坚持 local-first：优先本地导入、本地解析、本地存储。
- 规则层应集中维护，不要把 Patron、Variant、阵位限制等规则散落在页面组件里。
- 不要把第三方站点的私有 API、未公开接口或脆弱爬虫作为核心依赖；优先官方公开资料、可验证数据源和自维护结构化数据。

## 技术决策约束

- 现有“部署”和“数据存储”调研默认围绕 `Vite + React` 静态站展开；如果要改成其他框架，先确认是否需要同步修正文档和部署方案。
- 如果部署到 GitHub Pages 的 Project 站，必须处理好静态资源 `base` 路径。
- SPA 路由优先选择更稳妥的方案；如果没有明确理由，优先避免直接依赖服务端重写。
- 数据更新频率预计较低，新增或更新结构化数据时，优先维护 `public/data` 一类的版本化目录，并同步版本号文件。

## 代码与数据组织建议

- 先拆清核心实体：`Champion`、`Adventure`、`Variant`、`FormationLayout`、`UserProfile`、`RecommendationTemplate`。
- 公共数据、规则层、UI 层分开管理，避免页面直接承载原始规则判断。
- 如果新增脚本，优先服务于数据校验、数据转换、版本管理和规则测试，而不是一次性临时脚本堆积。
- 如果后续引入推荐逻辑，先做“规则过滤 + 模板推荐 + 可解释评分”，不要直接做黑盒结论。

## 变更时的同步要求

- 只要技术路线、目录结构、部署方案或数据方案发生实质变化，就同步更新相关调研文档，避免 AGENTS 与文档脱节。
- 新增工程骨架后，应补一份项目级 `README.md`，说明启动、构建、数据目录和部署方式。
- 修改数据结构时，尽量同时补字段说明或示例，避免后续 JSON 失控。

## 验证要求

- 文档改动至少保证内部表述一致，不要让 AGENTS、路线文档和部署文档互相冲突。
- 代码改动后，优先执行最小充分验证，例如构建、测试或数据校验脚本。
- 如果当前仓库还没有可运行项目，就明确说明“仅完成文档层变更，暂无可执行验证”。
