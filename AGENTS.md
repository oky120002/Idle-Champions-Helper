# Idle Champions 项目规范

## 产品边界

- 本仓库用于实现《Idle Champions of the Forgotten Realms》辅助网站，定位为个人成长导向的阵型决策台；第一阶段聚焦资料查询、限制筛选、阵型编辑、候选英雄校验和方案保存。
- 不做全自动最优阵容求解器或完整模拟器；优先建立可解释、可维护的决策闭环。
- 公共游戏数据采用版本化静态文件，个人数据坚持 local-first；如需推荐逻辑，应先做“规则过滤 + 模板推荐 + 可解释评分”，不直接输出黑盒结论。

## 技术与发布约束

- 技术路线固定为 `Vite + React + TypeScript`；未经过明确技术决策和文档同步，不切换主框架。
- 正式发布固定为 `GitHub Pages + GitHub Actions`；项目以零预算为前提，不引入付费托管、付费 API、付费 SaaS、付费构建服务或其他持续付费基础设施。
- GitHub Pages Project 站必须正确处理静态资源 `base` 路径，数据加载必须基于 `import.meta.env.BASE_URL`；MVP 阶段默认使用 `HashRouter`。
- 在没有用户明确要求前，不引入大体量后端工程、数据库或额外云基础设施。
- 除游戏基础数据获取外，第三方框架、工具链与服务必须开源或可免费合法使用；不把私有 API、未公开接口、规避限制的抓取方式或脆弱爬虫作为长期核心依赖。
- 数据更新频率较低时，结构化数据优先维护在 `public/data` 这类版本化目录，并同步版本号文件。

## 开发流程

- `main` 是主分支，必须长期与当前生产发布版本保持一致，不能用于日常开发。
- 日常开发必须在非 `main` 分支对应的工作树里进行，避免干扰其他并行任务。
- 工作树统一放在仓库外的工作树根目录 `~/.worktrees/Idle-Champions-Helper/<worktree-name>/`；
- 分支统一使用 `codex/` 前缀，允许 `codex/xxx` 或 `codex/xxx/xxx`。
- 需要新开工作树情况只有两类：当前在 `main`分支；或当前工作树分支与当前任务明显无关。只有在任务连续性明确、分支名高度相关时，才继续复用当前工作树。
- 开发与验证先在工作树内完成，再合并回本地 `main`；禁止把未合并、未验证完成的结果直接推到远端 `main`。
- 每次合并回 `main` 后，必须先在本地 `main` 执行一次面向发布前的全量回归；只有回归通过后，才允许推送远端 `main`，并由 `main` 的 push 触发 Actions 构建与发布。
- 若当前阶段尚未具备完整自动化测试体系，也必须执行当前可用的最大全量验证集合；仍有缺口时，先明确风险并获得用户确认。
- 与 GitHub 远端交互时（如查询 ref、推送、开 PR、改远端分支状态），默认优先使用 `gh` / `gh api`，不要先假设原生 `git push` / `git fetch` / `git ls-remote` 稳定可用；只有确认 Git 直连链路正常时，才回退到原生 `git` 网络命令。

## 质量与结构

- 默认采用 TDD（测试驱动开发）流程：先测试，后实现，再重构。
- 模块、规则、数据、组件与基础设施边界应高内聚、低耦合；核心规则、状态转换和副作用处理不得直接写死在页面层。
- 所有模块、功能与页面都应可测试；规则层、数据层、组件层与页面层需保持可隔离验证，并预留页面自动化测试能力，优先评估 `Playwright`。
- 新增脚本优先服务于数据校验、数据转换、版本管理和规则测试，而不是一次性临时脚本堆积。

## 文档与语言

- 用户可见文案、页面标题、导航、按钮、字段说明、README、`docs/` 文档和提交说明默认且绝对优先使用中文；只有没有稳定中文叫法的游戏术语、技术名词或第三方库名称才保留原文。
- 技术路线、目录结构、部署方案、数据方案、工程骨架、启动方式、构建方式或数据目录变化时，必须同步更新相关 `docs/` 与 `README.md`。
- `docs/` 目录结构与文档落位必须遵循 `docs/README.md`；除总索引外，不再把新文档直接平铺在 `docs/` 根目录。每个模块在进入开发前，必须先形成对应设计稿。
- 文档内引用仓库文件、目录、命令产物与代码位置时，统一使用**项目相对路径**（如 `docs/README.md`、`src/data/client.ts`）；禁止写绑定某个工作树、本机用户名或本地绝对目录的路径。只有在说明仓库外系统配置时，才允许写外部路径，并明确标注“这是仓库外路径”。
- 所有调研、检索、确认、查找类工作在形成结论后，都应沉淀为 `docs/` 下的 Markdown 文档，并保留结论、关键依据、确认时间与引用来源。
- 遇到开发流程、环境、认证、分支管理、发布等非代码缺陷问题时，解决后应主动同步到 `docs/troubleshooting-log.md`；单条记录原则上控制在 5~20 行，复杂问题尽量不超过 50 行，只写摘要和结论，不写流水账。
- 修改数据结构时，尽量同时补字段说明或示例；维护 `AGENTS.md` 时保持精简，若新增规则与既有条目冲突或需要取舍，先由用户裁决。

## 提交与验证

- 形成可独立识别、范围清晰的阶段性成功后，应自动执行一次独立 `commit` 固化成果；若工作区存在无关修改、用户未授权处理的既有变更或其他无法安全归并的改动，则不得强行提交。
- 同一提交只表达一个主目标，并选择最能代表该目标的前缀；提交信息统一使用 `xxx:xxxxxxxx`，优先使用 `feat`、`fix`、`docs`、`research`、`data`、`test`、`refactor`、`build`、`chore`。
- 文档改动必须保证 `AGENTS.md`、`README.md` 与相关路线、部署、数据文档表述一致；代码改动后优先执行最小充分验证，若存在缺口必须明确说明风险。
