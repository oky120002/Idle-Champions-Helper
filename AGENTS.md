# Idle Champions 项目规范

## 产品边界

- 本仓库用于实现《Idle Champions of the Forgotten Realms》辅助网站，产品定位为个人成长导向的阵型决策台；第一阶段聚焦资料查询、限制筛选、阵型编辑、候选英雄校验和方案保存。
- 不做全自动最优阵容求解器或完整模拟器；优先建立可解释、可维护的决策闭环。

## 发布、成本与合规约束

- 正式发布固定为 `GitHub Pages + GitHub Actions`，不将其他托管平台作为正式发布方案；项目以零预算为前提，不引入付费托管、付费 API、付费 SaaS、付费构建服务或其他持续付费基础设施。
- 除游戏基础数据获取外，第三方框架、库、工具链与服务必须开源或可免费合法使用；许可不明、试用受限、存在商业授权风险，或会给用户带来可预见法律、版权、平台条款或合规风险的方案默认不采用。
- 游戏基础数据获取不纳入前述法律、版权、平台条款与合规风险约束的评估范围。

## 语言与术语规范

- 用户可见文案、页面标题、导航、按钮、字段说明、README、`docs/` 文档、提交说明默认且绝对优先使用中文；游戏术语、技术名词或第三方库名称在没有稳定中文叫法时可保留原文，但**只要存在稳定中文叫法，就优先使用中文。**
- 代码标识符、第三方 API 字段名、库导出的类型或方法名按技术需要保留英文，不强行汉化。

## 实施原则

- 先数据模型与规则表达，再页面和视觉层；公共游戏数据采用版本化静态文件，个人数据坚持 local-first。
- 模块、规则、数据、组件与基础设施边界应遵循高内聚、低耦合原则：按职责拆分，跨层依赖最小化，避免双向依赖、隐式共享状态、横向耦合扩散，以及页面直接承载 Patron、Variant、阵位限制等核心规则。
- 即使为获取游戏基础数据，也不将私有 API、未公开接口、规避限制的抓取方式或脆弱爬虫作为长期核心依赖；应优先使用官方公开资料、可验证数据源和可沉淀为自维护结构化数据的来源。

## 技术决策约束

- 技术路线固定为 `Vite + React + TypeScript`；在没有明确重做技术决策并同步相关文档前，不切换主框架。
- 在没有用户明确要求前，不引入大体量后端工程、数据库或额外云基础设施。
- GitHub Pages Project 站必须正确处理静态资源 `base` 路径，并保证数据加载逻辑基于 `import.meta.env.BASE_URL`；MVP 阶段默认使用 `HashRouter`，避免依赖服务端重写或额外的 `404.html` 回退技巧。
- 数据更新频率较低时，结构化数据优先维护在 `public/data` 一类的版本化目录，并同步版本号文件。

## 测试与质量要求

- 默认采用 TDD（测试驱动开发）流程：先测试，后实现，再重构。
- 所有模块、功能与页面都必须可测试；规则层、数据层、组件层与页面层应按职责拆分，避免将关键规则、状态转换或副作用处理写成难以隔离验证的结构，并确保单元测试、集成测试与页面级测试可以分别落地。
- 页面自动化测试能力应作为标准能力预留，优先评估 `Playwright`；如后续采用其他自动化方案，需证明其同样满足开源或免费、可本地运行、可在 `GitHub Actions` 执行、适合页面回归测试等要求。

## 代码与数据组织约束

- 公共数据、规则层与 UI 层分开管理；新增脚本优先服务于数据校验、数据转换、版本管理和规则测试，而不是一次性临时脚本堆积。
- 如需引入推荐逻辑，应先做“规则过滤 + 模板推荐 + 可解释评分”，不要直接输出黑盒结论。

## 文档同步要求

- 技术路线、目录结构、部署方案、数据方案、工程骨架、启动方式、构建方式或数据目录发生变化时，应同步更新相关 `docs/` 与 `README.md`。
- 所有调研、检索、确认、查找类工作在形成结论后，都应沉淀为 `docs/` 目录下的 Markdown 文档；文件名统一采用语义明确的 kebab-case，并按“主题-类型”命名，例如 `static-hosting-research.md`、`champion-data-verification.md`；内容至少包含结论、关键依据、检索或确认时间，以及引用来源、链接或对应文件路径，便于后续复查与复用。
- 修改数据结构时，尽量同时补字段说明或示例，避免后续 JSON 失控。
- 维护 `AGENTS.md` 时应保持精简，优先合并语义相近条目；若新增规则与既有条目存在冲突、重复、歧义或需要取舍，应先提出待决问题，由用户裁决后再写入。

## 提交要求

- 形成可独立识别、范围清晰的阶段性成功后，应自动执行一次独立 `commit` 固化成果；提交粒度以单一清晰目标为准，优先选择最小可运行或最小可验证且边界完整的交付单元，但不得为了追求细粒度而把同一模块、同一功能或同一阶段内强相关的改动割裂成过碎提交。若工作区存在无关修改、用户未授权处理的既有变更或其他无法安全归并的改动，则不得强行提交，应先由用户裁决。
- 新模块或同一阶段内彼此强依赖的多项改动，可以在形成一个完整可运行或可验证单元后合并提交；缺陷修复、局部功能完善、独立测试补充、单篇调研或检索结论沉淀等边界清晰的工作，应在各自完成后分别提交。
- Git 提交信息统一使用 `xxx:xxxxxxxx` 格式；冒号前填写简短类型、范围或阶段标识，冒号后明确说明本次提交实际完成的对象、动作与结果，避免使用无法体现边界和成果的模糊描述。
- 提交前缀优先使用这些类型：`feat`（功能新增）、`fix`（缺陷修复）、`docs`（说明或规范文档）、`research`（调研与检索结论沉淀）、`data`（结构化数据新增或修订）、`test`（测试用例或测试基础设施）、`refactor`（不改变对外行为的重构）、`build`（构建、部署、工程配置）、`chore`（无法归入前述类型的例行维护）。
- 同一提交只表达一个主目标，并选择最能代表该主目标的前缀；避免使用 `update`、`misc` 一类语义空泛的提交前缀。

## 验证要求

- 文档改动必须保证 `AGENTS.md`、`README.md` 与相关路线、部署、数据文档表述一致。
- 代码改动后优先执行最小充分验证；若无法验证或当前无可执行项目，必须明确说明缺口与风险。
