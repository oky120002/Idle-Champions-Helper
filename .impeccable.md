## Design Context

### Users
- 主要用户是《Idle Champions of the Forgotten Realms》的活跃玩家，会在配队、查 seat 冲突、核对英雄技能 / 天赋 / 皮肤时打开这个站点。
- 这是一个偏个人决策辅助的资料站，核心场景是查资料、做限制筛选、编辑阵型、核对候选英雄与保存方案。
- 当前最核心的工作不是“看百科”，而是快速定位一个英雄的完整资料，并把这些信息转化为阵型判断。

### Brand Personality
- 理性、克制、战术化、可解释。
- 页面应该给人“资料很全，但不乱”的感觉，同时保留一点奇幻冒险游戏的金属感与徽记感。
- 情绪目标是降低信息焦虑、提升判断信心，而不是做成夸张炫技的展示页或通用 SaaS 控制台。

### Product UX Priorities
1. 先给结论性摘要，再给完整细节，最后给原始字段。
2. 高密度信息必须按问题分区，让用户一眼知道去哪找。
3. 任何“所有信息”都不能以牺牲扫描效率为代价；原始字段默认收起，但必须可达。
4. 页面内的重点交互应该围绕“筛选、比对、确认、保存”四种动作，而不是围绕装饰组织。
5. 公共层优先降低认知噪音，把视觉注意力留给结果卡、详情首屏和当前编辑对象。

## Current Visual Baseline

### Core Direction
- 当前唯一视觉基线是“低浮雕战术台”，不是旧版深材质卡片墙，也不是纯 flat。
- 整站继续使用深色主题，以铜金色和冷钢蓝做少量点缀，但共享层必须比过去更薄、更平、更克制。
- 目标是“共享层显著收平，重点内容区轻度保留深度”：大多数区域低材质，真正需要聚焦的对象只允许保留一层轻舞台感。
- 当前更高权重的视觉方向是“扁平高密度战术资料”：外层负责承载和聚焦，内部内容尽量去框、去背景、去阴影，用排版、间距、轻分隔线和少量徽记建立层级。
- 未上线产品按破坏性优化处理：不保留 legacy / compat / deprecated 样式分支，不做视觉双轨，不为旧风格留兜底。

### What Stays Flat
- 顶部壳层、导航、语言切换、背景光晕。
- 通用 panel、section、metric、badge、chip、button、input。
- workbench chrome、筛选区、统计区、辅助面板、formation 外壳。
- 高密度详情区的重复条目、同组对比列、升级轨道、属性块和说明块。
- 立绘页、宠物页、变体页、筛选页的页面级包装与说明层。

### What Keeps Light Depth
- 结果卡。
- 英雄详情页的 dossier / media 主区。

### Exception Rules
- 只保留 1 层主舞台背景 + 1 层轻阴影。
- 删除厚重发光、明显浮雕边、强内凹、重高光和大面积模糊。
- 删除嵌套卡片感：如果内容已经处在同一个 section / panel / workbench 容器里，内部同级项目默认不能再加完整背景框。
- 例外区承担“内容聚焦”职责，不能只是装饰更重。

## System Rules

### Shell And Navigation
- 顶部导航是悬浮指挥台，但现在必须是低浮雕版本：更扁、更紧、更居中、更接近工具带，而不是巨型按钮条。
- 导航按钮默认只做轻边框、轻底色、轻 active，不靠厚阴影和大面积高光制造层级。
- 导航图标如果存在，只能作为辅助；视觉重心应在文字、对齐和当前状态，而不是图标本身。
- 命中 workbench 路由时，桌面端头部默认保持紧凑态；页面内动作下放到工具条，不把页面级操作重新塞回顶部导航。
- 移动端顶部导航优先保证：触达高度、无横向溢出、紧凑收纳、打开后自然换行或分组展示，而不是依赖横向滚动条。

### Workbench Signature
- 桌面端页面签名固定为“上方小导航 + 下方单一工作台壳体”。
- 工作台外壳必须保持左右一体：左侧像前景抽屉，右侧像资料画布，但两侧 chrome 读起来是同一条工具带。
- 左工具条和左主体必须属于同一块抽屉；收起后左主体、边框和 gutter 一起退场，只保留紧凑锚点按钮。
- 页面不要重新长出独立的大页头、大副标题或额外 hero 卡，工作台本身就是页面级骨架。
- 工作台右上角 badge、toolbar action chips、结果区 compact metrics 应共享同一套低浮雕 chip 语法；不要一边是轻工具带，另一边又长出厚重按钮豆腐块。

### Surfaces And Controls
- 通用 surface 使用更薄的圆角、更弱的阴影、更浅的渐变、更轻的边框；默认看起来像“压平的资料面板”，不是厚卡片。
- 所有 chip、badge、button、input 默认更薄、更平、更密；hover 以轻提亮、轻描边、轻位移为主。
- 不再通过多层 inset、高饱和 glow、重复装饰线来堆层次。
- 页面私有样式优先只负责内容编排和少量例外；如果共享层已经能表达，就不要重复造一套材质系统。

### Flat Dense Detail Pattern
- 这是当前详情页和资料密集区的优先视觉语法，权重高于“给每个信息块一个卡片”的旧习惯。
- 一个信息组只保留一个外层容器；容器内部的并列列、总结、等级条目、说明段落默认透明，不再各自拥有背景框、厚边框或阴影。
- 层级通过 4 种低噪音手段表达：更明确的标题/摘要排版、稳定留白、1px 轻分隔线、少量数字或图形徽记。
- “总结 / 当前状态 / 所属分组”必须和“等级条目 / 明细行”有隔离感，但优先用标签语义、分隔线和留白隔离，不回退成厚卡片。
- 有短数字语义时优先用纯数字徽记，例如 seat `1`，不要写成长文本气泡；完整含义可放在 `aria-label` / `title` 中。
- 内部内容允许轻微放大来提升可读性，但不能靠重新加背景、发光、浮雕或卡片墙制造重点。

### Results And Headers
- 结果区头部不再重复页面已知的大标题、小标题；导航已说明页面语义时，不再重复“这是立绘图鉴/宠物图鉴/变体列表”这种废话。
- 结果区头部只承载实时统计、筛选摘要和必要状态，不承载高频操作；能上提到 toolbar 的，就不要塞回结果区里面。
- 可以保留动态统计，但应做成轻量实时指标，例如“当前展示 X / Y”，而不是厚重的标题区块。
- 高优先操作优先进入工作台 toolbar，尤其是“随机排序”“显示全部 / 收起到默认”这类结果级动作；结果区内部尽量不放按钮。
- 结果卡是整站最有层次的公共组件之一，但现在也必须瘦身：更轻的舞台、更轻的头像框、更克制的 spotlight。
- 立绘、宠物、变体等页面优先共享同一套结果卡语言，减少页面之间的视觉断层。

### Champion Detail
- 英雄详情页继续遵循“摘要 -> 分区 -> 原始字段”的卷宗结构。
- dossier / media 主区允许保留轻舞台感，但必须明显比旧版薄，重点放在跳读效率和首屏判断，而不是材质炫技。
- 徽记、标签、头像舞台等强调元素要服务于身份识别和信息定位，不能盖住正文扫描路径。
- 专精、能力、升级、天赋等详情子区优先套用“扁平高密度战术资料”模式：外层 section 是唯一舞台，内部重复条目透明化，使用轻分隔线和摘要标签区分组别。

### Formation UX
- 阵型页是复杂任务页，优先级是结构清晰、操作稳定、移动端可用，不是做成独立视觉特例。
- 布局库、棋盘外壳、移动端编辑器都跟随低浮雕基线收平；阵型本身的结构关系比外壳材质更重要。
- 移动端编辑必须继续保持单点聚焦，不因为收平风格而退化成功能删减版。

## Motion And Interaction

### Motion Baseline
- 动效统一使用“明显收敛”策略。
- hover 位移控制在 0~1px；不要再做明显上浮、弹跳、厚阴影跳变。
- 保留必要的结构型动效：抽屉开合、header 收紧、结果展开/收起、必要的滚动辅助。
- 动效主要传达结构关系和状态变化，不负责制造“高级感”。

### Interaction Contracts
- 所有页面默认不出现横向溢出；如果移动端信息多，优先换行、重排、折叠，而不是横向滑动。
- 按钮和切换控件优先保证命中面积与状态清晰度，再追求视觉紧凑。
- 顶部导航、section jump、filter chips 等高频控件必须保证点击目标稳定，不依赖精细像素操作。
- 工作台内的浮动返回顶部、筛选摘要、当前计数等辅助反馈必须轻量，不抢主内容注意力。

## Typography And Color
- 中文继续以 `PingFang SC` 负责正文扫描，以 `Songti SC` 负责标题与卷宗气质；不要退回通用 SaaS 字体组合。
- 色彩依然是深底 + 铜金 / 钢蓝点缀，但强调色必须稀缺，不把每个组件都染成重点。
- 中性色要带轻微品牌色偏，不使用死灰、纯黑、纯白。
- 文字强调靠层级、留白、对齐和字重，不靠渐变文字或彩色描边。

## Anti-Slop Guardrails
- 禁止回到默认后台审美：统一卡片墙、巨型彩色指标、圆角图标配标题、重复 hero metric 模板。
- 禁止渐变文字、粗侧边色条、无意义玻璃化、装饰性 sparkline、为了“质感”而增加的多层高光。
- 禁止页面头部重复讲已经由导航和上下文说明过的事情。
- 禁止为了兼容旧风格保留双轨类名、双轨 token 或“新旧都能过”的宽松实现。
- 不要把所有区域都做成重点；重点越少，真正的重点才越有效。

## Practical Defaults For Future Tasks
- 新页面先继承共享层低浮雕基线，再决定是否真的需要例外区深度。
- 新的资料密集区默认先套用“单一容器 + 内部去框 + 轻分隔”的扁平模式；除非用户明确要求更强舞台感，否则不要恢复嵌套卡片。
- 如果任务涉及导航、workbench、筛选区、阵型区，默认优先继续收平，而不是重新加厚。
- 如果任务涉及结果卡或详情首屏，可以保留轻深度，但先问自己：这是帮助聚焦，还是只是更花？
- 如果一个新说明文案只是重复页面语义，默认删掉；优先保留动态数量、状态、限制条件这类真正有信息价值的文本。
- 后续视觉迭代优先级：导航与壳层稳定性、workbench 一体感、结果卡多态一致性、详情页跳读效率、移动端结构稳定性。
