# 国内访问与静态托管备选方案调研

- 调研日期：2026-04-12
- 背景：Idle Champions 辅助站已确认采用 `Vite + React + TypeScript` 静态站路线，默认部署到 GitHub Pages，需要评估国内访问体验与后续备选托管方案。
- 当前结论：**第一阶段继续以 GitHub Pages 为默认方案；如果后续确认国内访问明显影响使用，再优先评估 Cloudflare Pages；只有在确实需要国内节点且愿意承担备案与运维成本时，再考虑国内云厂商。**

## 1. 先说结论

### 当前主方案不变

- 默认部署：`GitHub Pages + GitHub Actions`
- 默认域名：先用 GitHub Pages 提供的默认域名，避免额外备案与域名配置成本
- 当前理由：
  - 静态站足够适配本项目
  - 零服务端成本
  - 与 Git 工作流天然一致
  - 第一阶段重点仍然是数据、规则和页面闭环，不是部署体系

### 如果后续需要更好的国内访问

优先顺序建议如下：

1. **先评估 Cloudflare Pages**：仍然免备案，迁移成本相对低，全球分发能力通常好于 GitHub Pages 直连。
2. **再评估“GitHub Pages + CDN / 自定义域名”**：适合在不改主工作流的前提下优化访问体验。
3. **最后才评估国内云托管**：只有在确认用户主要来自中国大陆、且访问问题已经影响核心使用时才值得引入。

### 为什么不把国内托管当第一阶段默认值

- 中国大陆节点通常意味着 **ICP备案、实名、域名配置、运维细节** 一起上来。
- 免费额度、计费策略和产品形态变化较快，不适合作为第一阶段的稳定前提。
- 本项目当前是静态工具站，不值得一开始就为部署复杂度买单。
- 如果产品本身还没形成稳定使用场景，过早优化“国内极速访问”容易本末倒置。

## 2. 评估维度

为避免陷入“哪个平台宣传更便宜”的比较，后续统一按下面几个维度评估：

| 维度 | 关注点 |
| --- | --- |
| 免备案能力 | 是否可以直接用默认域名上线 |
| 静态站适配度 | 是否天然适配 `Vite + React + TypeScript` 产物 |
| Git 工作流兼容性 | 是否能通过 GitHub Actions 或仓库直连自动部署 |
| 国内访问体验 | 是否明显优于 GitHub Pages 直连 |
| 维护复杂度 | 是否需要额外 CLI、控制台操作、手动回源或手动刷新缓存 |
| 长期稳定性 | 免费额度、产品策略、账号要求是否容易变化 |

## 3. 候选方案判断

### GitHub Pages

**定位：默认主方案。**

优点：
- 与当前仓库天然一致
- 配置简单，适合静态站
- 不需要备案
- 能直接接入 GitHub Actions 工作流

限制：
- 中国大陆访问体验不一定稳定
- 对 SPA 来说需要自己处理 `base` 和路由策略
- 缓存和边缘控制能力有限

结论：**第一阶段继续使用。**

### Cloudflare Pages

**定位：第一优先级备用方案。**

优点：
- 免费层通常足够支撑小型静态工具站
- 不需要备案
- 直连 GitHub 仓库比较顺手
- 对静态资源分发和缓存控制更友好

限制：
- 仍然属于海外平台，国内访问体验虽常常比 GitHub Pages 更好，但不能等同于“国内托管”
- 如果后续接入自定义域名和复杂缓存策略，仍需要额外配置

结论：**如果 GitHub Pages 的访问体验不能接受，优先试这个。**

### 国内云厂商静态托管或对象存储网站托管

这里不再逐个平台比免费额度和单价，因为这类信息变化太快，且对第一阶段决策意义不大。统一按一类方案看待：

- 典型形态：对象存储静态网站托管、静态托管产品、对象存储 + CDN
- 典型平台：腾讯云、阿里云、七牛云、又拍云等

共性优点：
- 中国大陆访问稳定性通常更好
- 更容易接入国内 CDN、日志、监控和域名体系

共性限制：
- 自定义域名通常离不开备案
- 配置项更多，运维复杂度更高
- 免费额度和价格规则经常变化，需要以官方控制台为准
- 对当前这个阶段的项目来说，收益未必大于复杂度

结论：**不是第一阶段默认方案，只作为后续明确需求后的升级路线。**

## 4. 备案判断

### 什么时候要把备案算进成本里

| 场景 | 是否默认要考虑备案 |
| --- | --- |
| GitHub Pages 默认域名 | 否 |
| Cloudflare Pages 默认域名 | 否 |
| 海外平台 + 自定义域名 | 一般不需要大陆备案，但要看解析与合规要求 |
| 国内云平台 + 自定义域名 | 通常要把备案当成前置条件 |

### 备案带来的实际影响

- 上线时间会拉长
- 域名、主体、实名认证都要同步处理
- 后续变更域名或迁移平台时也会受影响
- 如果只是验证 MVP，不值得提前引入

## 5. 分阶段部署建议

### 阶段一：MVP / 工程骨架阶段

- 方案：`GitHub Pages + GitHub Actions`
- 域名：默认域名
- 目标：先把查询、限制筛选、阵型编辑、保存闭环做出来

### 阶段二：开始有真实用户反馈

- 观察项：
  - 中国大陆打开速度是否明显拖慢使用
  - 静态 JSON 数据加载是否成为瓶颈
  - 是否出现高频反馈“打不开”或“加载太慢”
- 如果需要优化：先试 `Cloudflare Pages` 或 CDN 加速

### 阶段三：确认大量中国大陆访问且体验必须稳定

- 再评估国内托管
- 决策前先确认：
  - 是否愿意处理备案
  - 是否需要自定义域名
  - 是否真的比“Cloudflare Pages / CDN 加速”更划算

## 6. 当前决策记录

- **技术路线确认：`Vite + React + TypeScript`**
- **默认部署确认：`GitHub Pages + GitHub Actions`**
- **国内访问优化优先级确认：`Cloudflare Pages` 高于国内云托管**
- **国内云厂商方案当前状态：保留为后续备选，不进入第一阶段实现范围**

> 当前最重要的不是把部署方案做到最复杂，而是先把站点本身做成一个值得访问的工具。
