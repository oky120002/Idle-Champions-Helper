# 国内托管：决策、候选方案与备案边界

- 日期：2026-04-12
- 目标：回答“为什么第一阶段不改主方案”“哪些候选平台值得考虑”“备案什么时候才需要算进成本”。

## 当前主方案不变

- 默认部署：`GitHub Pages + GitHub Actions`
- 默认域名：先用 GitHub Pages 提供的默认域名，避免额外备案与域名配置成本
- 当前理由：静态站足够适配本项目、零服务端成本、与 Git 工作流天然一致、第一阶段重点仍然是数据、规则和页面闭环，而不是部署体系

## 如果后续需要更好的国内访问

优先顺序建议如下：

1. 先评估 Cloudflare Pages：仍然免备案，迁移成本相对低，全球分发能力通常好于 GitHub Pages 直连。
2. 再评估“GitHub Pages + CDN / 自定义域名”：适合在不改主工作流的前提下优化访问体验。
3. 最后才评估国内云托管：只有在确认用户主要来自中国大陆、且访问问题已经影响核心使用时才值得引入。

## 评估维度

统一按这些维度评估：免备案能力、静态站适配度、Git 工作流兼容性、国内访问体验、维护复杂度、长期稳定性。

## 候选方案判断

### GitHub Pages

默认主方案；优点是与当前仓库天然一致、配置简单、适合静态站、不需要备案；限制是中国大陆访问体验不一定稳定、SPA 需要自己处理 `base` 和路由策略、缓存和边缘控制能力有限。

### Cloudflare Pages

第一优先级备用方案；优点是免费层通常够用、不需要备案、直连 GitHub 仓库顺手、对静态资源分发和缓存控制更友好；限制是仍属海外平台，国内访问体验虽常常比 GitHub Pages 更好，但不能等同于“国内托管”。

### 国内云厂商静态托管或对象存储网站托管

共性优点是大陆访问稳定性通常更好，更容易接入国内 CDN、日志、监控和域名体系；共性限制是自定义域名通常离不开备案、配置项更多、运维复杂度更高、免费额度与价格规则变化快。结论：不是第一阶段默认方案，只作为后续明确需求后的升级路线。

## 备案判断

- GitHub Pages 默认域名：默认不考虑备案
- Cloudflare Pages 默认域名：默认不考虑备案
- 海外平台 + 自定义域名：一般不需要大陆备案，但要看解析与合规要求
- 国内云平台 + 自定义域名：通常要把备案当成前置条件
