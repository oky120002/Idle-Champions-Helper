# 方案存档模块设计稿

- 设计日期：2026-04-13
- 模块目标：提供本地优先的阵型方案、常用筛选组合和阶段性目标存档能力，承接“查到候选 -> 摆出阵型 -> 保存复用”的闭环。
- 当前结论：第一阶段先围绕阵型方案存档做 MVP，不把它扩成完整任务管理器或云同步系统。

## 模块定位

- 方案存档模块是产品闭环里的“沉淀层”。
- 前面的页面解决“查什么英雄、看什么限制、怎么摆阵型”；这一页解决“当前这套阵容要不要留下、下次能不能快速找回来、这套方案是为哪种目标准备的”。
- 它本质上是个人工作台的本地记录层，不是展示页占位。

## 第一阶段范围

| 可保存对象 | 第一阶段结论 | 原因 |
| --- | --- | --- |
| 阵型方案 | MVP 先做 | 与阵型编辑页衔接最直接，数据结构最清晰，也最容易验证复用价值 |
| 常用筛选组合 | 后续再做 | 先不要把三套模型一起做完 |
| 阶段性目标清单 | 后续再做 | 避免模块膨胀成任务管理器 |

额外约束：阵型页负责“最近草稿”，方案存档页负责“已命名方案”；未命名草稿不进入方案列表。

## MVP 数据模型

| 字段 | 说明 |
| --- | --- |
| `id` | 本地唯一标识 |
| `schemaVersion` | 方案结构版本 |
| `dataVersion` | 保存时对应的公共数据版本 |
| `name` | 方案名 |
| `description` | 备注或用途说明 |
| `layoutId` | 对应阵型布局 |
| `placements` | 槽位到英雄的映射 |
| `scenarioRef.kind` | `campaign` / `adventure` / `variant` / `trial` / `timeGate` |
| `scenarioRef.id` | 用于恢复规则上下文的正式场景身份；手工联调方案可为 `null` |
| `scenarioTags` | 例如“推图 / 速刷 / 试炼”，只作为用户可读标签，不作为恢复主键 |
| `priority` | `low / medium / high`，仅作为个人排序辅助 |
| `createdAt` / `updatedAt` | 创建与最近编辑时间 |

卡片里如果需要 `seat` 摘要，应在读取方案时由 `placements + champions` 运行时推导，而不是把它作为长期持久化源字段。

## 存储策略

- 正式方向：本地优先，正式持久化目标是 `IndexedDB`。
- 开发阶段建议两步走：先用内存 mock 跑通页面结构和卡片流转，再补 `IndexedDB` 适配层接正式持久化。
- 不建议为了过渡再补一套长期 `localStorage` 方案，否则后面会多出迁移和兼容成本。

## 加载校验与回退

方案恢复时必须先校验：`dataVersion`、`layoutId`、`scenarioRef`、`slotId`、`championId` 是否仍可识别。

| 场景 | 建议行为 |
| --- | --- |
| 保存版本可读且全部有效 | 正常打开与编辑 |
| 保存版本可读但部分失效 | 保留有效部分；明确标记“该方案含失效引用”；允许用户进入阵型页后重新保存 |
| 保存版本不可读 | 使用当前版本尝试兼容恢复，并明确提示“该方案基于新版本兼容恢复” |
| 大范围失效 | 阻止静默恢复；提示用户复制备注后删除旧方案 |

原则：优先按方案保存时的 `dataVersion` 恢复；只有旧版本目录不可读时才退到“当前版本兼容恢复”，且不能伪装成原样恢复。

## 页面结构建议

1. 顶部说明卡：说明“本地优先，不上传”，说明这一页主要保存什么。
2. 方案列表区：最近编辑、空态提示、方案卡片。
3. 操作区：新建方案、从当前阵型导入、编辑名称与备注、删除方案。

第一阶段不需要复杂分页；优先把空态、列表态、编辑态跑通。

## 与其他模块的衔接

- 来自阵型编辑模块的核心输入：`layoutId`、`placements`、`scenarioRef`、当前数据版本 `dataVersion`。
- 来自英雄筛选模块的后续扩展：当前筛选条件快照、进入方案详情时反向恢复候选池。
- 来自个人数据模块的后续扩展：标记该方案里哪些英雄未拥有、根据个人画像提示替代位；这一步要等个人画像结构稳定后再接。

## 核心交互建议

MVP 至少支持：查看方案列表、创建方案、编辑方案名与备注、删除方案、从阵型草稿生成方案、识别并提示失效引用方案。

最小流程：阵型编辑页形成草稿 -> 点击“保存为方案” -> 填写名称 / 备注 / 场景标签 -> 保留当前 `scenarioRef` -> 写入本地存储 -> 方案存档页可见并可再次打开。

## 验收标准

- 用户能把至少一套阵型草稿保存到本地。
- 刷新页面后仍能看到已保存方案。
- 已保存方案再次打开时，能恢复对应布局与场景上下文，或明确提示当前方案没有绑定场景。
- 空态、列表态、删除态都明确可用。
- 公共数据版本变化后，旧方案不会静默坏掉。
- 数据结构可以继续承接筛选快照和个人画像扩展。
