# 方案存档模块设计稿

- 设计日期：2026-04-13
- 模块目标：为用户提供本地优先的阵型方案、常用筛选组合和阶段性目标存档能力，承接“查到候选 -> 摆出阵型 -> 保存复用”的闭环。
- 当前结论：第一阶段先围绕阵型方案存档做 MVP，不把它扩成完整任务管理器或云同步系统。

---

## 1. 模块定位

方案存档模块是整个产品闭环中的“沉淀层”。

前面的页面解决：

- 查什么英雄
- 看什么限制
- 怎么摆阵型

这一页要解决：

- 当前这套阵容要不要留下
- 下次还能不能快速找回来
- 这个方案是为哪种目标准备的

所以它本质上是“个人工作台”的本地记录层，而不是展示页占位。

---

## 2. 第一阶段范围

第一阶段只做三类可保存对象中的第一类：

1. 阵型方案
2. 常用筛选组合
3. 阶段性目标清单

其中 MVP 只先落：

- 阵型方案存档

原因：

- 它和阵型编辑页衔接最直接
- 数据结构最清晰
- 最容易验证是否真的能帮助重复使用

同时明确：

- 阵型页负责“最近草稿”
- 方案存档页负责“已命名方案”

未命名草稿不进入方案列表，避免把临时编辑状态和正式方案库混在一起。

---

## 3. 数据模型建议

### 3.1 MVP 方案模型

```ts
interface ScenarioRef {
  kind: 'campaign' | 'adventure' | 'variant' | 'trial' | 'timeGate'
  id: string
}

interface FormationPreset {
  id: string
  schemaVersion: 1
  dataVersion: string
  name: string
  description: string
  layoutId: string
  placements: Record<string, string>
  scenarioRef: ScenarioRef | null
  scenarioTags: string[]
  priority: 'low' | 'medium' | 'high'
  createdAt: string
  updatedAt: string
}
```

字段说明：

- `id`：本地唯一标识
- `schemaVersion`：方案结构版本
- `dataVersion`：保存时对应的公共数据版本
- `name`：方案名
- `description`：备注或用途说明
- `layoutId`：对应阵型布局
- `placements`：槽位到英雄的映射
- `scenarioRef`：用于恢复规则上下文的正式场景身份；当前手工联调方案可为 `null`
- `scenarioTags`：例如“推图 / 速刷 / 试炼”，只作为用户可读标签，不作为恢复主键
- `priority`：仅作为个人排序辅助
- `createdAt`：方案创建时间
- `updatedAt`：排序和最近编辑时间

卡片中如果需要 `seat` 摘要，应在读取方案时由 `placements + champions` 运行时推导，而不是把它作为长期持久化源字段。

### 3.2 后续再加的对象

后续可以增加：

- `SavedChampionFilter`
- `GoalNote`

但不要在 MVP 阶段一次把三套模型全做完。

## 4. 存储策略

### 4.1 正式方向

- 本地优先
- 正式持久化目标：`IndexedDB`

这与当前仓库对个人数据的总方向保持一致，不额外再引入服务端或云数据库。

### 4.2 开发阶段建议

可以按两步走：

1. 先用内存 mock 跑通页面结构和卡片流转
2. 再补 `IndexedDB` 适配层接正式持久化

注意：

- 不建议为了过渡再补一套长期 `localStorage` 方案
- 否则后面会多出一层迁移和兼容成本

### 4.3 加载校验与回退策略

方案恢复时，必须先校验：

- `dataVersion` 是否仍可识别
- `layoutId` 是否仍存在
- `scenarioRef` 若存在，其对应场景 / 规则上下文是否仍可识别
- `slotId` 是否仍属于对应布局
- `championId` 是否仍能被当前数据解析

恢复语义应与阵型草稿保持一致：

1. 优先按方案保存时的 `dataVersion` 读取对应版本目录
2. 只有在旧版本目录不可读时，才退回“当前版本兼容恢复”
3. 兼容恢复必须显式提示，不允许伪装成原样恢复

建议行为：

1. 保存版本可读且全部有效
   - 正常打开与编辑
2. 保存版本可读但部分失效
   - 保留有效部分
   - 明确标记“该方案含失效引用”
   - 允许用户进入阵型页后重新保存
3. 保存版本不可读，转兼容恢复
   - 使用当前版本尝试映射
   - 明确标记“该方案基于新版本兼容恢复”
4. 大范围失效
   - 阻止静默恢复
   - 提示用户复制备注后删除旧方案

因此，方案页后续同样需要依赖“按指定版本读取集合”的数据访问能力。

## 5. 页面结构建议

建议按下面结构组织：

1. 顶部说明卡
   - 说明“本地优先，不上传”
   - 说明这一页主要保存什么
2. 方案列表区
   - 最近编辑
   - 空态提示
   - 方案卡片
3. 操作区
   - 新建方案
   - 从当前阵型导入
   - 编辑名称与备注
   - 删除方案

第一阶段不需要复杂分页；优先把空态、列表态、编辑态跑通。

## 7. 核心交互建议

MVP 至少支持：

- 查看方案列表
- 创建方案
- 编辑方案名与备注
- 删除方案
- 从阵型草稿生成方案
- 识别并提示失效引用方案

推荐的最小流程：

```text
阵型编辑页形成草稿
  ↓
点击“保存为方案”
  ↓
填写名称、备注、场景标签
  ↓
保留当前 `scenarioRef`
  ↓
写入本地存储
  ↓
方案存档页可见并可再次打开
```

---

## 8. 验收标准

第一阶段完成后，应满足：

- 用户能把至少一套阵型草稿保存到本地
- 刷新页面后仍能看到已保存方案
- 已保存方案再次打开时，能恢复对应布局与场景上下文，或明确提示当前方案没有绑定场景
- 空态、列表态、删除态都明确可用
- 公共数据版本变化后，旧方案不会静默坏掉
- 数据结构可以继续承接筛选快照和个人画像扩展

---

## 9. 当前明确不做

- 不做云同步
- 不做账号系统
- 不做公开分享库
- 不做协作编辑
- 不做复杂目标管理或提醒系统
