# 共享组件模块设计稿

- 设计日期：2026-04-13
- 模块目标：治理跨页面高度重复、语义稳定的 UI / 国际化实现，优先减少页面层散落逻辑，同时避免为了“去重”而过度抽象。
- 当前结论：当前已完成两轮治理——第一轮沉淀双语展示组件；第二轮确认“状态条”和“字段块”适合继续共享化，而结果卡整体仍暂不抽取。

## 1. 模块定位

共享组件模块负责承接“已经跨页面重复、并且继续增长概率很高”的实现，不负责：

- 单一页面内的一次性排版片段
- 语义还不稳定、后续大概率会推翻的临时封装
- 只为了少写两三行 JSX 的过早抽象

当前项目里，页面数量已经开始增加，国际化显示逻辑和结果卡片结构都在扩散，所以需要一个稳定的治理入口。

## 2. 本轮抽取依据

### 已确认的重复模式

2026-04-13 在代码中确认到的高重复国际化模式：

1. 双语主副文本展示
   - `src/pages/ChampionsPage.tsx`
   - `src/pages/VariantsPage.tsx`
   - `src/pages/FormationPage.tsx`
   - 共同模式都是：先取当前语言主文本，再决定是否展示另一语言副文本
2. `seat` 的界面语言格式化
   - `src/pages/ChampionsPage.tsx`
   - `src/pages/FormationPage.tsx`
   - `src/pages/PresetsPage.tsx`
3. 列表内“仅跟随当前界面语言显示主文本”
   - 联动队伍按钮
   - 变体限制条件
   - 变体奖励

### 本轮决策

- 抽组件：`LocalizedText`
  - 原因：它承接的是稳定的“展示行为”，而不是单个页面的布局偶然性。
- 抽组件：`StatusBanner`
  - 原因：同一套 `tone + title/detail + actions` 结构已在多个页面反复出现，且页面继续增加时还会持续增长。
- 抽组件：`FieldGroup`
  - 原因：`label + 控件 + hint` 的字段块在表单与筛选区都已稳定复现，适合统一结构和语义。
- 抽 helper：`formatSeatLabel`
  - 原因：这是纯字符串格式化，没有必要为了一个短标签引入额外 DOM 层。
- 暂不抽组件：结果卡整体、优先级按钮组
  - 原因：结果卡整体当前字段差异仍偏大；优先级按钮组仍是页面私有业务交互，直接抽会把页面差异硬塞进共享 props。

## 3. 组件抽取准入规则

满足下面至少两项，才建议进入共享组件候选：

1. 同一语义已出现在至少 2 个页面，且总重复点不少于 3 处。
2. 页面继续扩张时，重复逻辑还会自然增长。
3. 可以用稳定、可解释的 props 命名，而不是页面临时变量名。
4. 能明确写出测试矩阵，覆盖主路径和关键边界。

出现下面任一情况时，先不要抽：

- 只是视觉相似，但交互或数据语义不同。
- 还不清楚“通用 API”应该长什么样。
- 抽出来后会让调用方额外写更多样板代码。
- 当前需求只在一个页面存在，且短期没有复用证据。

## 4. 开发流程规范

每次新增共享组件，统一遵循下面流程：

1. 先查组件索引
   - 开发前必须先看 `docs/modules/shared-components/shared-components-catalog.md`
   - 如果已有共享组件或 helper 已能覆盖当前场景，必须直接复用，不得在页面内重复实现等价结构
   - 只有在现有组件无法覆盖、并且能明确说清新增收益时，才允许继续新增页面私有实现或新的共享组件
2. 先盘点重复点
   - 记录重复位置、语义是否一致、哪些地方故意不抽。
3. 先写测试再实现
   - 先覆盖主路径、语言切换、空值/边界场景，再开始改代码。
4. 组件只承接稳定能力
   - 页面私有布局、业务分支、临时文案不要塞进共享组件。
5. 接入页面后跑回归
   - 至少覆盖：新增组件测试 + 受影响页面既有测试 + lint/typecheck。
6. 同步更新文档
   - 必须更新 `docs/modules/shared-components/shared-components-catalog.md`
   - 必须视情况更新本设计稿里的经验沉淀

## 5. 测试规范

共享组件默认需要两层测试：

1. 纯函数 / helper 测试
   - 适合 `formatSeatLabel` 这类纯格式化逻辑
2. 组件行为测试
   - 验证不同语言、不同模式、不同包装元素、边界文本是否都符合预期

本项目当前对共享组件的最低测试要求：

- 至少覆盖一种默认场景
- 至少覆盖一种语言切换场景
- 至少覆盖一种“副文本应消失”的边界场景
- 至少覆盖一种可配置渲染方式，例如包装元素或分隔符

## 6. 文档治理规范

为了让后续智能体和开发人员能先查再写，文档同步遵循以下规则：

- 开发前先查组件目录；命中可用组件或 helper 时，默认必须复用
- 如果决定不复用已有共享资产，必须写清原因，例如语义不一致、现有 API 无法覆盖、强行复用会增加复杂度
- 可直接复用的组件，收录到 `shared-components-catalog.md`
- 只停留在候选阶段、还没稳定下来的模式，不进入组件目录，只在设计稿里记录
- 每次组件 API 变化时，同步更新：用途、核心 props、示例、使用位置
- 如果某个组件被证明抽取收益不高，应在文档里明确写“回退原因”，避免后续重复踩坑

## 7. 首轮经验沉淀

### 国际化不要混成一个概念

当前仓库至少有两类国际化文本：

- UI 文案：`{ zh, en }`，由 `useI18n().t()` 处理
- 游戏数据双语字段：`{ original, display }`，由 `LocalizedText` 处理

两类文本来源不同、回退规则不同，不应强行共用同一个组件 API。

### “可复用组件”不等于“所有重复都抽组件”

`seat` 标签重复很多，但它更适合做纯 helper；如果硬抽成组件，只会增加无意义 DOM 和调用噪音。

### 组件目录和文档索引必须一起长

共享组件如果没有统一索引，后续智能体很容易继续在页面里重新造轮子。组件代码和组件目录文档必须同步维护，不能只做其一。

### 结果卡要按“整卡复用”而不是“局部长得像”来判断

本轮重新筛过结果卡后，结论仍是暂不抽整卡组件。虽然标题区结构有相似点，但卡片正文的字段组合、状态提示、操作区和标签区差异仍然明显；如果现在硬抽，只会得到一个 props 过多、可读性变差的壳组件。

### 英雄展示要先看现成的展示组件族

在并行开发里，`ChampionAvatar`、`ChampionIdentity`、`ChampionPill` 已经自然形成了一组“英雄展示组件”：

- `ChampionAvatar` 负责头像资源、`alt` 文案和回退占位
- `ChampionIdentity` 负责卡片标题区的身份展示
- `ChampionPill` 负责紧凑列表里的单行英雄标签

后续页面如果只是重复出现“头像 + 名字 + seat / 上下文标签”的组合，优先复用这组组件，而不是重新在页面里拼 `img`、双语名称和回退逻辑。只要这类组件进入 `src/components/` 并开始跨页面复用，就必须同步登记到组件索引，并补齐对应测试。
