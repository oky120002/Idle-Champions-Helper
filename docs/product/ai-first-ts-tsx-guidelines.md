# AI-first TS/TSX 开发规范

- 方案日期：2026-04-17
- 适用范围：`src/**/*.ts`、`src/**/*.tsx`、`tests/**/*.ts`、`tests/**/*.tsx` 以及相关配置文件
- 目标：在“100% AI 开发、强调渐进式加载、需要节约 token”的前提下，为仓库建立稳定、低上下文成本、可局部修改的 TypeScript / React 规范

## 1. 设计目标

- 规范第一目标不是统一表层风格，而是降低 AI 的理解成本、修改风险和上下文体积。
- 常规需求默认应能通过读取少量入口文件完成；避免为了改一处而扫描整个 feature。
- 新代码优先保证可局部理解、可局部测试、可局部替换；不要依赖隐式耦合和跨层回跳。
- 工具规则优先约束正确性、边界清晰度和可维护性；格式争议不进入高优先级规则。

## 2. 通用共识与仓库策略

### 2.1 国际高共识

- TypeScript 以严格类型检查为基线，`strict` 是默认起点。
- `unknown` 优先于 `any`；边界输入必须先收窄再使用。
- Promise 不能悬空；联合类型分支需要穷尽处理。
- React 组件与 Hook 保持纯净；Hook 只在顶层调用；Effect 只用于与外部系统同步。
- 类型导入与值导入要明确区分，避免运行时与类型空间混淆。

### 2.2 本仓库的 AI-first 加严策略

- 命名导出优先，禁止 `default export`。
- 新代码默认优先使用 `type`；只有在“公共对象契约确实需要可扩展”时才使用 `interface`。
- 页面组件只做编排，不承载大型规则、状态机、映射表或长链式数据变换。
- 禁止新增泛化兜底文件：`utils.ts`、`helpers.ts`、`common.ts`、`misc.ts`。
- Barrel 只允许局部、稳定、低扇出的聚合；禁止跨 feature 的“全量出口文件”。

## 3. 渐进式加载优先

### 3.1 读取顺序

- AI 进入一个 feature 时，优先按下面顺序加载：
  1. 入口页面或入口组件：`FeaturePage.tsx`
  2. 状态 Hook：`useFeatureState.ts`
  3. 区块组件：`sections/*.tsx`
  4. 纯逻辑：`model.ts`
  5. 类型定义：`types.ts`
  6. 常量与文案：`constants.ts`
- 若一个 feature 超过 5 个核心文件，允许增加一个 40-60 行的局部 `README.md`，只写入口、状态来源、主要子模块和不变量。

### 3.2 结构约束

- 一个文件只承载一个主职责；一个目录只承载一个明确子域或一个 feature。
- 纯逻辑放 `.ts`，UI 组合放 `.tsx`，副作用与状态编排放 `useXxx.ts`。
- 大型映射表、静态选项、标签集合、文案常量必须拆到邻接 `.ts` 文件，不要塞在页面中。
- 不在 JSX 中直接书写复杂变换；先在组件顶部或 `.ts` 模块中命名中间结果。
- 不在组件函数体内定义大型类型、长常量、长工具函数；应前移到局部模块。

## 4. 文件体量预算

- 新增 `.ts`：单一职责文件在 300 行内默认可保留；超过 300 行应评估拆分；超过 450 行应拆；超过 700 行必须拆。
- 新增 `.tsx`：单一职责文件在 250 行内默认可保留；超过 250 行应评估拆分；超过 350 行应拆；超过 500 行必须拆。
- 页面级入口 `.tsx`：硬上限可放宽到 400 行，但前提是区块组件、Hook、模型层已经外置。
- 这些阈值服务于 token 节约，而不是追求更小文件：TS / TSX 语义密度高、状态和数据流集中，单文件过大时会显著抬高一次读取成本。
- 是否继续拆分，优先看“常见任务能否用更少文件完成”而不是只看行数；如果继续拆会让一次常见修改必须额外多开文件，就先保留当前边界。
- 长字符串、大型静态映射、schema、生成代码可按职责适度豁免，但仍要保持单一职责和可测试性。
- 存量超大文件按“触碰即减债”处理：本次改动涉及它时，至少顺手拆出一层 `sections`、`model` 或 `types`。

## 5. TypeScript 规则

- 启用 `strict`、`noUncheckedIndexedAccess`、`exactOptionalPropertyTypes`、`noImplicitOverride`、`noImplicitReturns`。
- 边界数据、`catch` 错误、第三方输入统一从 `unknown` 开始，不直接假设结构正确。
- 状态优先使用判别联合，不要用多个布尔值或多个可空字段拼接状态机。
- 常量映射优先 `as const` + `satisfies`，减少类型漂移与重复声明。
- 类型断言 `as` 只允许出现在边界适配、第三方接入或显式校验之后。
- 新代码默认用 `type`；已有 `interface` 可保留，除非本次改动正好能低成本重构。

## 6. TSX / React 规则

- 页面组件负责路由参数、顶层状态 Hook、区块编排和 props 透传，不直接实现大型规则。
- 每个复杂交互优先抽成一个 Hook；每个可独立折叠的视觉区块优先抽成一个组件。
- Effect 只用于外部同步；能在 render 阶段直接计算的内容不要放进 Effect。
- 异步事件处理必须显式兜底错误；不要依赖隐式吞错。
- 组件 props 和本地状态名使用领域词汇，不使用空泛名词如 `data`、`info`、`handler2`。

## 7. 注释与文档

- 少写注释，优先通过命名和结构表达语义。
- 注释只用于三类信息：非显然业务约束、状态机/副作用边界、兼容性陷阱。
- 局部设计说明优先落在邻近文档或局部 `README.md`，不要把长说明写进组件顶部。
- 长注释、重复注释和叙述式注释会增加 token 负担，应主动删除或下沉。

## 8. 工具落地

- `tsconfig*.json` 负责严格类型边界，优先拦截不安全访问、遗漏返回和可选属性误用。
- `eslint.config.js` 负责类型导入、悬空 Promise、无效条件、分支穷尽、命名导出等 AI-first 硬约束。
- 对现有存量债务，lint 允许使用 `warn` 标出体量问题，但正确性问题必须保持 `error`。
- 工具规则不追求把所有代码风格都编码化，只编码那些能显著降低误改风险的约束。

## 9. 迁移策略

- 先收紧配置，让新代码和新改动默认进入新轨道。
- 再按“当前改到哪里，就顺手把哪里降上下文成本”的原则逐步拆旧文件。
- 页面超大文件优先抽出 `sections/`、`useXxx.ts`、`model.ts`；公共类型再下沉到 `types.ts`。
- 如果某条规则会导致当前大面积阻塞，先保留文档约束和低噪音 lint，再分阶段升级为更强约束。
