# 文档治理方案

- 日期：2026-04-16
- 目标：让 `AGENTS.md`、`README.md` 和 `docs/` 各司其职，减少重复、过时内容和上下文浪费。

## 单一事实源

- `AGENTS.md`：只写仓库级硬规则和长期稳定约束。
- `README.md`：首页；只保留项目现状、常用命令、高频入口。
- `docs/README.md`：`docs/` 总入口；只保留导航、放置规则和更新触发条件。
- `docs/*/README.md`：主题局部索引；承接渐进式加载。
- 专题文档：规则、调研、设计、排查的唯一展开位置。

## 低 token 写法

- 先写结论、边界、当前状态，再写依据；不写长引导语。
- 入口文档只导航，不展开专题细节；长清单按主题下沉到局部索引。
- 仓库里已有单一事实源时，文档优先引用路径，例如 `vite.config.ts`、`.github/workflows/deploy.yml`、`src/data/client.ts`，不重复粘贴整段 YAML / JSON / 代码。
- 同一事实只在一处展开；其他位置只保留一句摘要和链接。
- 历史问题只保留可复用结论，并明确仍有效、已失效或已归档。

## 渐进式加载

- 目录规则或入口：读 `docs/README.md`。
- 产品范围与阶段目标：读 `docs/product/README.md` 及其指向文档。
- 外部事实、数据或技术选型：读 `docs/research/README.md` 及对应主题文档。
- 模块上下文：读 `docs/modules/README.md` 后再进目标模块。
- 排查和历史验证：读 `docs/investigations/README.md` 或 `docs/troubleshooting-log.md`。

## 触发更新

- 改了脚本、命令、部署链路、路由策略、数据目录或核心范围。
- 新增或删除文档，或发现多个入口重复维护同一事实。
- 某目录文档数量增长到需要局部索引，或原局部索引失效。
- 文档出现已失效命令、错误路径、绝对路径、会话专用信息或与代码冲突的描述。

## 更新动作

- 先改单一事实源，再改入口摘要和索引。
- 根 `README.md` 与 `docs/README.md` 只补最小必要变化，不重新堆专题细节。
- 新增文档簇时，先补对应目录 `README.md`，再扩专题文档。
- 规则、调研、排查都要写清适用边界，必要时标日期和“当前是否仍有效”。
