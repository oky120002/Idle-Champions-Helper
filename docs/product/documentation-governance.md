# 文档治理方案

- 方案日期：2026-04-16
- 目标：让 `AGENTS.md`、`README.md` 和 `docs/` 各司其职，减少重复、过时内容和上下文浪费。
- 当前结论：`AGENTS.md` 只保留项目级硬约束，`README.md` 只保留首页信息，`docs/README.md` 只保留总入口，专题细节下沉到各目录自己的 `README.md` 或专题文档。

## 1. 单一事实源

- `AGENTS.md`：只写本仓库差异化、长期稳定的硬规则；不重复全局通用沟通、真实性和工具使用规则。
- `README.md`：面向仓库访问者的首页，只保留项目是什么、怎么跑、怎么验证、去哪继续读。
- `docs/README.md`：`docs/` 总入口，只负责导航、放置规则和拆分策略。
- `docs/product/README.md`、`docs/research/README.md`、`docs/modules/README.md`、`docs/investigations/README.md`：各主题局部索引，承接渐进式加载。
- 具体规则、调研、设计和排查结论继续留在最贴近事实的专题文档。

## 2. 精简原则

- 同一事实只在最合适的一处展开，其他入口只保留导航或一句摘要。
- 优先写稳定约束、工作流和入口，不写容易过时的长清单。
- 与上层全局规则重复的内容不再抄回仓库文档；仓库内只补项目差异和落地约束。
- 能按主题拆开的内容不要继续堆回 `AGENTS.md`、`README.md` 或 `docs/README.md`。
- 历史问题只保留可复用结论，不在高频入口重复长过程。

## 3. 渐进式加载

- 默认只读取当前任务直接相关的文件，不预加载整棵 `docs/`。
- 需要目录规则或入口时，读 `docs/README.md`。
- 需要产品范围与全局方案时，读 `docs/product/README.md` 及其指向文档。
- 需要外部事实、数据或技术选型时，读 `docs/research/README.md` 及对应主题目录。
- 需要模块上下文时，读 `docs/modules/README.md` 再进入目标模块目录。
- 需要排查或历史验证时，读 `docs/investigations/README.md` 或 `docs/troubleshooting-log.md`。

## 4. 触发更新

- 改了脚本、命令、部署链路、路由策略或数据目录。
- 改了项目范围、核心页面、模块职责或关键非目标。
- 新增或删除文档，或发现多个入口在重复维护同一事实。
- 某个目录的文档数量增长到需要局部索引，或原有局部索引已经失效。
- 文档出现已失效命令、错误路径、会话专用绝对路径或和代码冲突的描述。

## 5. 更新动作

- 先改单一事实源，再改高频入口的链接和摘要。
- 根 README 和 `docs/README.md` 只补最小必要变化，不把专题清单重新堆回首页。
- 新增一个文档簇时，优先先补对应目录的 `README.md`，再继续扩展明细文档。
- 规则、调研、排查类结论都要写清适用边界，必要时标注日期。
