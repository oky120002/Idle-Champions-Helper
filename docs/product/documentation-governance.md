# 文档治理方案

- 日期：2026-04-16
- 目标：让 `AGENTS.md`、`README.md` 和 `docs/` 各司其职，并把所有主题都纳入渐进式加载，减少重复、过时内容和上下文浪费。

## 单一事实源

- `AGENTS.md`：仓库级硬规则和长期稳定约束
- `README.md`：项目现状、常用命令、高频入口
- `docs/README.md`：`docs/` 总入口；只保留导航、放置规则和更新触发条件
- 目录级 `README.md`：主题局部索引；承接按需加载
- 子主题 `README.md`：大主题继续下钻时的第二层导航
- 叶子文档：规则、调研、设计、排查的唯一展开位置

## 低 token 写法

- 先写结论、边界、当前状态，再写依据
- 高频、稳定、可预判命中的规则直接指向叶子文档；只有低频、分叉或目标不明确的主题才通过 `README.md` 索引分流
- 任何规则或阈值调整，都必须以降低单次任务的总 token 成本为前提；优先减少“需要多开几个文件才能完成一次修改”的情况
- 父索引只列主题，不列全量子文档；长清单下沉到更深一层 `README.md`
- 同一事实只在一处展开；其他位置只保留一句摘要和路径
- 历史问题只保留可复用结论，并标明仍有效、已失效或已归档
- 能用表格或短列表表达的，不用长段解释；能指向代码的，不重复粘整段代码
- 仓库里已有单一事实源时，只引用路径，例如 `vite.config.ts`、`.github/workflows/deploy.yml`、`src/data/client.ts`

## 强制渐进式加载

默认读取顺序必须是：根入口 -> 目录 README -> 子主题 README -> 叶子文档。

适用规则：

- 一个主题如果同时包含“概览 + 决策 + 实现 + 验证 + 审计”，必须拆开，不允许继续堆在同一篇里
- 目录一旦长到需要解释“先看哪篇”，就补 `README.md`
- 子主题一旦增长到多个叶子文档，再补子主题 `README.md`
- 入口文档只回答“去哪读”和“什么时候读”，不承担长篇细节

## 目标体积

以下是治理目标，不是硬编码 lint：

- 根 / 目录 `README.md`：尽量控制在“几十秒能扫完”的长度，优先 200-800 字级别
- 子主题入口页：尽量控制在约 300-1200 字级别
- 叶子文档：优先保持在单一问题内；如果已经明显跨越多个问题、多个读者意图，或出现 5 个以上 `##` 主段落，就应继续拆分
- 任何文档只要让人产生“为了看第 1 个问题，被迫吞下另外 3 个问题”的情况，就说明还应该再拆

## 拆分维度

优先按读者意图拆，而不是平均切字数：

- 概览 / 入口
- 约束 / 决策
- 实现 / 操作
- 验证 / 审计
- 案例 / 经验库
- 引用 / 证据

## 触发更新

- 改了脚本、命令、部署链路、路由策略、数据目录或核心范围
- 新增 / 删除文档，或发现多个入口重复维护同一事实
- 某目录或主题重新膨胀，导致需要加载过多无关内容
- 文档出现失效命令、错误路径、绝对路径、会话专用信息或与代码冲突的描述

## 更新动作

- 先改单一事实源，再改入口摘要和索引
- 根 `README.md` 与 `docs/README.md` 只补最小必要变化，不重新堆专题细节
- 新增文档簇时，先补对应目录 `README.md`，再扩叶子文档
- 规则、调研、排查都要写清适用边界，必要时标日期与有效性
