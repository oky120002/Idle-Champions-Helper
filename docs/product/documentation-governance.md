# 文档治理方案

- 方案日期：2026-04-13
- 目标：让 `README.md`、`docs/`、排查记录和模块设计持续保持及时性、正确性、逻辑性；遇到过期、冲突、重复或环境耦合内容时，明确该修复、合并、降级入口还是删除。
- 当前结论：**`docs/README.md` 是 `docs/` 的单一总索引；凡是技术路线、命令、部署、数据、页面、验证结论发生变化，都要触发一次文档扫描。**

## 1. 治理目标

- 事实可回溯：关键结论都能回指到代码、配置、脚本、命令输出或外部来源。
- 入口可扫描：读者能先从仓库首页和 `docs/README.md` 找到正确入口，而不是在重复清单里来回对照。
- 历史与现状分层：保留历史排查价值，但不能让历史结论伪装成当前事实。
- 环境细节可移植：优先写仓库相对路径、脚本名、工作流名和验证步骤，避免把一次会话的绝对路径写成长期契约。

## 2. 文档分工与单一事实源

- `AGENTS.md`：记录仓库级硬约束、流程规则和长期稳定规范，不承载操作细节流水账。
- `README.md`：只放项目现状、常用命令和高频入口；完整清单不再在这里重复维护。
- `docs/README.md`：`docs/` 唯一总索引，负责目录规则、放置判定和当前文档入口说明。
- `docs/product/`：产品边界、路线图、全局方案，包括这份文档治理方案。
- `docs/research/`：调研与选型结论，必须写清结论、依据、时间和边界。
- `docs/modules/`：模块设计稿与模块级验证，描述目标、范围、数据合同、验收标准。
- `docs/investigations/`：排查、验证、运行环境记录；若已成为历史问题，要明确标记“历史归档”或“当前是否仍有效”。
- `docs/troubleshooting-log.md`：非代码问题的摘要台账，只记录可复用结论和入口，不重复长过程。

## 3. 触发扫描时机

- 改了 `package.json` 脚本、构建链路、测试命令、工作流或发布方式。
- 改了路由、页面入口、导航、核心功能范围或数据目录结构。
- 改了 `vite.config.ts`、数据加载路径、`import.meta.env.BASE_URL` 用法或本地预览方式。
- 排查结论发生变化，例如“原来失败、现在已修复”或“原来是当前问题、现在只剩历史归档价值”。
- 新增或删除文档，尤其是 README、模块设计稿、专题调研和长期排查记录。
- 发现文档里出现会话专用绝对路径、已经失效的命令、重复索引或相互冲突的描述。

## 4. 单次扫描清单

### 4.1 入口与结构

- `README.md` 的高频入口是否仍然成立。
- `docs/README.md` 当前索引是否覆盖所有现存文档。
- 新文档是否放在正确目录，而不是直接平铺在 `docs/` 根目录。

### 4.2 事实与命令

- 文档里的脚本名、工作流名、页面入口、数据文件和目录结构是否和仓库一致。
- 涉及验证结果的文档，是否标明验证时间、命令和关键结论。
- 涉及外部来源的文档，是否保留来源链接、确认时间与适用边界。

### 4.3 历史与现状

- 排查类文档是否清楚说明“当前仍有效”还是“历史归档”。
- 设计类文档是否把“已实现”和“待实现”分开描述。
- 已经被新方案替代的旧文档，是否应降级入口、合并摘要或删除。

### 4.4 环境耦合

- 是否写入了机器专用绝对路径、临时目录、一次性截图目录或会话上下文。
- 若这些信息只是辅助说明，应改成仓库相对路径、脚本名或“本地临时目录”这类稳定表述。

## 5. 修复、合并、删除的判定

- 修复：文档仍有价值，但事实、命令、路径或状态已经漂移。
- 合并：两个入口重复解释同一件事，且分开维护只会制造冲突；优先保留离事实最近、范围最合适的那一份。
- 降级入口：文档仍值得保留，但只剩历史参考价值，应从高频入口移除，并在索引里标明历史属性。
- 删除：内容已完全被新文档覆盖，且没有保留历史原理或引用价值；删除前先确认没有其他文档继续引用它。

## 6. 执行约束

- 用户可见文档默认使用中文；术语确实没有稳定中文叫法时再保留原文。
- 研究、检索、确认、排查类文档必须写时间与依据，不能只写结论。
- 涉及命令或运行结果时，只写已验证过的内容；未验证的假设要明确标出。
- 优先减少重复清单和重复定义，避免同一事实在多个入口展开维护。
