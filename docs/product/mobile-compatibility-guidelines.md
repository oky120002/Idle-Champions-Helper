# 移动端兼容性开发规范

- 生效日期：2026-04-14
- 当前结论：移动端默认禁止横向滑动作为主要交互；尤其不允许把主导航、筛选项、按钮组或关键信息列表做成需要左右拖动才能正常使用的形态。

---

## 1. 适用范围

以下场景默认都受本规范约束：

- 顶部主导航
- 筛选区与筛选结果快捷入口
- 标签组、按钮组、切换组
- 主要信息卡列表
- 首屏或长页面里的 sticky 区域

---

## 2. 强制规则

1. 移动端页面默认不得出现“为了访问主要功能而必须横向滑动”的交互。
2. 移动端页面默认不得出现页面级横向溢出；验收时应确认 `document.documentElement.scrollWidth` 不大于 `clientWidth`。
3. 主导航、筛选和主操作在移动端空间不足时，优先改为：
   - 折叠面板
   - 抽屉
   - 分步展开
   - 纵向分组
   - 更少但更明确的常驻入口
4. sticky 区域在移动端必须优先压缩常驻内容；不能把完整桌面导航、长说明和大段副标题一起长期固定在顶部。

---

## 3. 默认替代方案

当移动端空间不足时，默认按以下优先级处理：

1. 先减少常驻信息量
2. 再把低频入口折叠起来
3. 再把并列入口改成纵向分组或两列网格
4. 最后才考虑分页、二级入口或更深层级导航

明确不作为默认方案的做法：

- 横向滚动导航
- 横向滑动筛选条
- 必须左右拖动才能看到完整按钮组
- 依赖“用户自己发现还能横向滑”的隐藏式交互

---

## 4. 例外边界

只有在以下内容天然属于横向画布时，才允许讨论横向交互：

- 棋盘 / 阵型画布
- 时间轴
- 大表格
- 明确以横向浏览为核心的媒体内容

即便如此，也应同时满足：

- 先有用户明确认可
- 页面提供非横向滑动的替代入口或摘要
- 不影响主任务闭环

---

## 5. 当前落地说明

本规范直接来源于 2026-04-14 的移动端头部体验修正：

- 问题：顶部 sticky 区域在移动端同时固定品牌、说明、语言切换与 6 个一级导航，导致页面内容长期被遮挡。
- 废弃方案：把一级导航做成单行横向滑动。
- 当前采用：紧凑头部 + 折叠式主导航面板，展开后使用纵向网格，不依赖横向滑动。
