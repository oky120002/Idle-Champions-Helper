# Idle Champions 辅助网站调研与实施路线

- 研究日期：2026-04-11
- 项目目标：围绕《Idle Champions of the Forgotten Realms》构建一个以“资料查询 + 阵型编排 + 英雄筛选 + 成长决策”为核心的辅助网站，帮助玩家更快完成清图、刷资源、推进成长循环。
- 当前结论：这个项目值得做，但不建议一开始就做“全自动最优解网站”。更稳妥的路线是先做“个人决策台”，把查询、过滤、阵型搭建、限制校验、个人进度映射做扎实，再逐步扩展到推荐与优化。

---

## 1. 这件事为什么值得做

《Idle Champions》不是单纯的英雄收集游戏，它的核心复杂度来自多个系统叠加：

1. 英雄位（seat）冲突与阵型站位限制。
2. 冒险/变体/Patron/Trials/Time Gate 等模式对英雄可用性的限制。
3. 装备、传奇、feats、pigments、favor、blessings、modron 等成长系统互相影响。
4. 玩家真正关心的不是“单个英雄强不强”，而是“我现在这局/这条线/这个限制下，怎么组最省时间”。

这意味着玩家需要的不是单一百科，而是一个“可组合、可过滤、可验证、可保存、最好还能和个人账号状态关联”的工具平台。

如果产品做得对，价值会明显高于纯资料站：

- 查询效率更高：快速找到某个限制下能上的英雄与可行组合。
- 决策成本更低：减少频繁切游戏、查 wiki、查社区表格、查第三方站点的来回切换。
- 个人相关性更强：同样一套攻略，对不同玩家未必适用；你的站点如果能结合“我拥有什么、我缺什么、我正在推什么”，价值会大幅提升。

## 2. 调研结论摘要

### 2.1 现有优秀站点已经证明需求存在

你给出的两个参考站点已经覆盖了大量玩家痛点：

- [Byteglow](https://ic.byteglow.com/) 更偏“以个人数据为中心的工具台”，覆盖 champions、formation、events、patrons、trials、modron、variants、inventory、legendaries、speed、summary 等多个页面。
- [Kleho](https://idle.kleho.ru/) 更偏“深度数据查看器 + 规则/地图/活动辅助工具”，覆盖 adventures、maps、hero、favors、patrons、timegates、modron-auto、leg-finder 等页面。

这说明：

- 市场需求不是“要不要做”，而是“你做成什么差异化价值”。
- 单纯复制现有站点意义不大；你需要明确自己优先服务哪类决策场景。

### 2.2 最有机会的方向不是“更全”，而是“更贴近实际推图决策”

参考站点已经很全，但仍有一个常见缺口：

- 数据很丰富。
- 玩家真正临场决策时，仍要自己判断“现在该上谁、下谁、换哪套、哪条成长线最值得优先”。

所以你的机会不是从零重做一个百科，而是做一个更偏“行动建议”的站：

- 以目标为中心，而不是以数据表为中心。
- 以当前可用英雄/限制条件为输入，而不是默认玩家全都懂。
- 以阵型与清图效率为主线，而不是只做静态页面罗列。

### 2.3 数据能力是成败关键，不是前端页面数量

从参考站点的前端实现和公开页面可以推断，社区工具的核心不在于页面，而在于三件事：

1. 游戏 definitions / 基础资料的持续更新。
2. 用户个人数据的导入、解析、映射。
3. 把“规则 + 限制 + 阵型 + 个人拥有情况”组合起来的计算层。

这里最难的是第 3 点。也就是说，这个项目最先要啃的是“数据结构和规则引擎”，不是视觉层。

> 注：这一条是基于参考站点公开页面与前端脚本行为做出的技术推断，不等同于官方文档声明。

## 3. 参考网站拆解

### 3.1 Byteglow：个人数据驱动的综合工具台

#### 我观察到的优点

- 页面模块非常完整，覆盖查询、阵型、Patron、Trials、Speed、Modron、Legendaries、Variants 等多个核心决策场景。
- 明确支持用户数据配置，并强调“数据只保存在浏览器本地，不上传到其他地方”，这对玩家隐私与信任很重要。
- 能围绕“玩家自己的账号状态”来做视图和过滤，而不是只展示通用资料。
- 更像“玩家控制台”，而不是纯百科站。

#### 值得借鉴的做法

- 本地优先（local-first）的用户数据导入与存储思路。
- 按“任务场景”拆工具页，而不是只按“数据类型”分栏目。
- 允许用户保存阵型、目标、筛选条件、偏好设置。

#### 不建议直接照搬的部分

- 页面功能非常多，如果你一开始就照这个规模做，项目很容易失控。
- 对新玩家来说，信息密度偏高，学习成本不低。
- 工具入口多，但如果没有统一的“任务流”，用户还是需要自己拼装步骤。

### 3.2 Kleho：深度数据视图与规则资源站

#### 我观察到的优点

- 在 adventures、maps、hero、patrons、favors、timegates 等方面的数据深度很强。
- 很适合做“查限制、查地图、查活动、查具体条目”的工作。
- 有比较明显的数据视图和工具化路线，比如 map、modron-auto、timegates、leg-finder 等，都指向具体问题。
- 更像“专业资料库 + 规则分析器”。

#### 值得借鉴的做法

- 把冒险、地图、限制、奖励、Patron 任务等拆成结构化数据而不是纯文本。
- 让页面既能做浏览，也能做筛选和定位。
- 针对特定高频问题做小工具，而不是只做一个大而全页面。

#### 不建议直接照搬的部分

- 数据视图导向很强，但如果没有个人进度映射，用户还是要把站内信息再翻译成自己的行动方案。
- 对一般玩家来说，可能更像专家工具，而不是“我现在该怎么打”的助手。

### 3.3 你应该如何定位自己

推荐不要在第一阶段试图打败这两个站点的“全面性”，而是建立自己的差异化定位：

**定位建议：个人成长导向的阵型决策台。**

一句话版本：

> 帮玩家在“当前拥有条件 + 当前目标 + 当前限制”下，更快得到可执行的阵容和成长决策。

这个定位比“又一个数据库站”更有机会。

## 4. 建议的产品范围

### 4.1 核心用户画像

优先服务这两类玩家：

#### A. 中度玩家

特征：

- 已经理解基本玩法，但系统越来越多，开始经常查资料。
- 想知道 Patron、Time Gate、Trials、Variants 到底该怎么配队。
- 想减少“反复试错”的时间。

诉求：

- 快速知道当前限制下哪些英雄可用。
- 快速摆阵。
- 快速知道缺口在哪。

#### B. 成长型玩家 / 回坑玩家

特征：

- 英雄池和装备不完整。
- 想更快建立过渡阵容与刷资源策略。
- 对“先补谁、先推哪条线、先开哪类系统”更敏感。

诉求：

- 明确成长优先级。
- 给出当前阶段能执行的建议，而不是最终毕业阵。

不建议第一阶段优先服务：

- 极少数顶层硬核玩家的“极限数学最优解”。
- 社区社交、发帖、评论、排行榜这类平台化需求。

### 4.2 产品边界

第一阶段不要做：

- 全自动最佳阵容求解器。
- 覆盖所有玩法与所有特殊规则的完整模拟器。
- 用户系统、社区系统、内容运营系统。
- 自己维护一套和全网完全一样的大百科页面矩阵。

第一阶段必须做：

- 结构化数据底座。
- 可过滤的英雄/冒险/限制查询。
- 可用阵型编辑器。
- 基于限制条件的候选英雄筛选。
- 阵容保存与比较。

## 5. 推荐的实施顺序

### 5.1 Phase 0：项目定义与数据摸底（1 周）

目标：先确认“做什么”和“数据从哪来”，避免一上来写站。

要做的事：

1. 列出核心实体：Champion、Seat、Role、Affiliation、Patron Eligibility、Adventure、Variant、Formation Layout、Buff/Tag、Gear/Feat、User Ownership、Preset。
2. 定义第一版用户任务：
   - 我想查某个 seat 可上的核心英雄。
   - 我想在某个 Patron/Variant 限制下筛出可上阵英雄。
   - 我想针对某个目标（速刷 / 推图 / Trials / Time Gate）搭一个可保存阵容。
3. 确认数据来源策略：官方公开信息、游戏 definitions、用户本地导入、人工补录。
4. 画出第一版信息架构图和页面树。
5. 先做低保真原型，不写复杂业务代码。

交付物：

- 数据字段清单。
- 页面地图（sitemap）。
- MVP 范围说明。
- 风险清单。

验收标准：

- 你能用一句话说清网站第一版只服务哪些决策场景。
- 你能列清楚哪些数据必须有，哪些可以后补。

### 5.2 Phase 1：数据底座（1-2 周）

目标：把后续所有页面都依赖的数据模型立住。

要做的事：

1. 建立 champions 数据表：seat、roles、tags、affiliations、availability、patron eligibility、关键技能摘要。
2. 建立 adventures / variants 数据表：campaign、目标区间、限制条件、奖励、解锁关系。
3. 建立 formation layout 数据：不同战役/活动/试炼阵型槽位与前后排关系。
4. 建立 normalized rules 表达方式：把“只能上 X”、“不能上 Y”、“必须满足 Z”转成结构化规则。
5. 建立数据更新脚本和版本号记录。

交付物：

- 可持续更新的数据仓。
- 数据导入脚本。
- 基础管理脚本或内部检查页。

验收标准：

- 能通过脚本稳定生成前端可消费的 JSON/API 数据。
- 新增英雄、活动、限制后，不需要手工改一堆页面。

### 5.3 Phase 2：MVP 查询站（1-2 周）

目标：先上线“好用的查询 + 过滤”，证明工具价值。

优先页面：

1. 英雄页：按 seat、role、tag、affiliation、Patron、模式限制做筛选。
2. 冒险/变体页：查看限制、奖励、推荐阵容思路占位。
3. 阵型页：显示阵型槽位、拖拽/点击放置英雄、检测 seat 冲突。
4. 组合候选页：输入当前目标，输出可尝试的候选英雄池。

MVP 不追求：

- 自动算出最优阵。
- 全量成长推荐。
- 精准伤害模拟。

验收标准：

- 你自己能在站里完成一次“查限制 -> 选英雄 -> 摆阵 -> 保存阵容”的闭环。

### 5.4 Phase 3：个人化能力（2 周）

目标：让网站从“通用工具”升级为“你的工具”。

要做的事：

1. 支持用户导入个人数据，优先采用本地解析、本地存储思路。
2. 标记 owned / unowned champions、装备等级、可用 feats、favor、patron 进度等。
3. 在筛选结果中增加“我能不能用”“我还差什么”的提示。
4. 支持保存个人阵容模板、目标清单、常用筛选方案。

验收标准：

- 同一个冒险页面，对两个不同账号能给出不同的候选集。
- 用户不上传账号隐私到你的服务器也能获得核心价值。

### 5.5 Phase 4：决策辅助与推荐（2-4 周）

目标：从“信息展示”提升到“行动建议”。

建议先做三类建议：

1. 速刷建议：优先偏 speed/quest reduction/自动推进相关组合。
2. 推图建议：偏 BUD / push / 生存 / debuff 组合。
3. 限制模式建议：Patron、Trials、Time Gate、Variant 下的可行阵容骨架。

实现方式不要一开始就追求 AI 或复杂优化器，优先做：

- 规则过滤。
- 评分卡。
- 模板推荐。
- 人工维护的阵容 archetype 库。

验收标准：

- 用户输入目标后，能得到 3-5 套“可解释”的候选方案，而不是一串黑盒结果。

### 5.6 Phase 5：高级功能（后续）

可以放到后面再做：

- Time Gate / Patron / Trials 的更细致路线规划。
- Modron / 多队自动化辅助。
- 缺口分析（还差哪些英雄、装备、feats 才能组成某套体系）。
- 成长路线建议（下一个优先解锁/投资目标）。
- 社区分享链接、公开阵容库、评价体系。

## 6. 我建议你的 MVP 长什么样

### 6.1 一句话 MVP

**一个可以按限制筛英雄、按阵型排阵容、保存方案、并逐步接入个人账号数据的工具站。**

### 6.2 MVP 页面建议

| 页面 | 目标 | MVP 内容 |
| --- | --- | --- |
| 首页 | 让用户快速开始 | 进入查询、进入阵型、进入目标模式 |
| 英雄查询 | 找到可用英雄 | seat/role/tag/affiliation/Patron/模式过滤 |
| 冒险/变体查询 | 看限制和目标 | campaign、variant、reward、限制规则 |
| 阵型编辑器 | 摆阵并校验 | 阵位图、seat 冲突提示、保存草稿 |
| 候选推荐 | 给出第一轮可尝试方案 | 基于目标模式和限制的候选英雄池 |
| 个人面板（后加） | 让数据变成“我的数据” | owned champions、缺口、已保存阵容 |

### 6.3 MVP 成功指标

先不要用大而虚的“日活”做目标，建议看：

- 你自己是否愿意把它作为主要查询入口。
- 是否能显著减少切换多个网站/表格/截图的次数。
- 是否能在 1 分钟内完成一次阵容筛选与保存。
- 是否能明显降低“这个限制到底能上谁”的查找时间。

## 7. 技术路线建议

### 7.1 技术原则

- **先数据，后界面。**
- **先规则过滤，后自动推荐。**
- **先本地优先，后云同步。**
- **先可维护，后炫技。**

### 7.2 推荐架构

#### 前端

建议使用：

- Next.js + TypeScript
- React Query
- Zustand 或 Jotai
- Tailwind CSS 或你熟悉的组件体系
- 拖拽库用于阵型编辑（如 dnd-kit）

理由：

- 页面型工具站适合 React 生态。
- 后续如果需要 SSR、静态生成、SEO、分享链接，Next.js 比较稳。
- 阵型编辑和复杂筛选会需要较强的前端状态管理。

#### 数据层

建议使用：

- PostgreSQL 存结构化业务数据。
- Prisma 之类 ORM 做模型管理。
- 定时导入脚本维护 champions / adventures / variants / formations / tags。

如果第一版想更轻，也可以：

- 先用版本化 JSON 文件做只读数据源。
- 等规则稳定后再迁移到数据库。

#### 规则层

这是最关键的一层，建议独立出来：

- 输入：冒险限制、Patron 条件、目标模式、用户拥有情况。
- 输出：可用英雄池、冲突提示、推荐骨架、缺口分析。

不要把规则散落在前端页面里，应该集中管理，便于维护和测试。

### 7.3 数据来源建议

建议分成四层：

1. **官方公开资料层**：官网、公告、活动说明，用于校验系统名称、模式名称、版本变化。
2. **游戏定义数据层**：用于英雄、冒险、限制、装备、任务等结构化底座。
3. **用户本地导入层**：用于 owned champions、装备、favor、目标进度等个人化能力。
4. **人工补充层**：用于“推荐阵容 archetype”“成长建议”“常见打法说明”。

这里最重要的一条：

**不要把你的核心能力建立在第三方站点的私有 API 上。**

原因：

- 不稳定。
- 难维护。
- 法律和伦理边界不清晰。
- 一旦对方改动，你会被动。

更稳妥的方式是：

- 用可验证的数据来源搭自己的底座。
- 第三方站点只用于竞品分析，不作为核心依赖。

## 8. 你最需要先设计的数据模型

建议第一版至少包含这些实体：

### Champion

- id
- 名称
- seat
- roles
- affiliations
- tags / race / alignment / gender 等限制相关字段
- patron eligibility
- 获取方式
- 关键机制标签（speed / gold / support / dps / tank / debuff / click 等）

### Adventure / Variant

- id
- campaign
- 名称
- 目标区域
- 解锁条件
- Patron 可用性
- 限制条件（结构化）
- 奖励
- 对应阵型布局

### Formation Layout

- layout id
- 槽位坐标
- 前后排关系
- 可放置数量
- 适用战役/模式

### User Profile

- 已拥有英雄
- 装备/传奇/feats 状态
- favor / blessings / patron 进度
- 常用阵型预设
- 目标偏好（速刷/推图/试炼等）

### Recommendation Template

- 场景类型
- 核心英雄
- 替代英雄
- 使用前提
- 不适用条件
- 操作说明

## 9. 风险与应对

| 风险 | 说明 | 应对 |
| --- | --- | --- |
| 数据源不稳定 | 游戏更新频繁，字段可能变化 | 给导入脚本做版本记录与校验 |
| 规则太复杂 | 各模式限制叠加后容易失控 | 先只覆盖高频规则，逐步扩展 |
| 范围膨胀 | 很容易变成“再做一个 Byteglow/Kleho” | 明确 MVP 只做决策闭环 |
| 个人数据敏感 | 用户会担心账号数据 | 本地解析、本地存储、明确提示 |
| 推荐不可信 | 黑盒建议会失去信任 | 先做可解释推荐，不做神秘分数 |
| 维护成本高 | 新英雄、新活动持续增加 | 数据模型和规则引擎先行，页面只是视图 |

---

## 10. 我给你的明确建议

如果你现在就要开始，我建议按下面顺序推进：

1. **先定 MVP，不要定“大而全网站”。**
2. **先做数据结构和规则表达，不要先写 UI。**
3. **先做“限制筛选 + 阵型编辑 + 保存方案”闭环。**
4. **随后接入个人数据，做出差异化价值。**
5. **最后再做推荐、优化、成长路线。**

换句话说：

- 第一步不是“怎么做漂亮网站”，而是“怎么把游戏规则表达成机器可处理的数据”。
- 第二步不是“收集所有页面”，而是“完成一个真正能帮助你推图的闭环”。
